AgentForce CRM

Enterprise Salesforce DX application repository fully orchestrated with CumulusCI (CCI), automated GitHub Actions scratch org provisioning, Apex testing, and Robot E2E UI automation.

Salesforce API 66.0 CumulusCI 4.10.0 GitHub Actions Automated

Overview & Architecture

The AgentForce CRM architecture combines standard Salesforce DX source format (`force-app`) with CumulusCI (CCI) flow orchestration and GitHub Actions automation.

⚡
Salesforce DX
Organized using standard `force-app` metadata format targeting API Version 66.0.
☁️
CumulusCI Orchestration
Configured via `cumulusci.yml` to automate org creation, metadata deployment, and profile setup.
🤖
GitHub Actions Workflow
On-demand scratch org creation with custom admin email, password generation, and markdown export.

Dev Hub Authentication & Secret Setup

Before running automated scratch org workflows on GitHub Actions, complete the one-time authentication between Salesforce Dev Hub and GitHub Secrets.

Prerequisite CLI Tools
Ensure you have Salesforce CLI (`sf`), CumulusCI (`cci`), and GitHub CLI (`gh`) installed on your workstation.

Step 1: Authenticate Dev Hub Locally

Bash 
sf org login web --set-default-dev-hub --alias MyDevHub

Step 2: Export SFDX Auth URL & Set GitHub Secret

Bash 
# 1. Retrieve the SFDX Auth URL credential string
sf org auth show-sfdx-auth-url --target-org MyDevHub --json

# 2. Store the secret in your GitHub repository using GitHub CLI (gh)
gh secret set DEVHUB_AUTH_URL --body "force://PlatformCLI::<your_copied_auth_url>"

Step 3: How to Switch / Change Dev Hub Orgs

To point GitHub Actions to a new Dev Hub org in the future, simply authenticate the new org and update the `DEVHUB_AUTH_URL` secret:

Bash 
sf org login web --set-default-dev-hub --alias NewDevHub
sf org auth show-sfdx-auth-url --target-org NewDevHub --json
gh secret set DEVHUB_AUTH_URL --body "force://PlatformCLI::<new_auth_url>"

GitHub Actions Scratch Org Workflow

This repository includes a manual GitHub Actions workflow ([`.github/workflows/create-scratch-org.yml`](./.github/workflows/create-scratch-org.yml)) that provisions a scratch org using CumulusCI (`feature_org` flow), configures a custom admin email, generates the admin user password, and outputs full credentials into `scratch_org_credentials.md`.

Running the Workflow

Option A: Via GitHub CLI (Command Line)

Bash 
gh workflow run create-scratch-org.yml -f adminEmail="developer@company.com"

Option B: Via GitHub Web UI

  1. Navigate to your GitHub repository ➔ Actions tab.
  2. Select Create Scratch Org with CumulusCI.
  3. Click Run workflow, type your email into Admin Email for the scratch org, and click Run workflow.

Dynamic Admin Email Injection

Before scratch org creation, the workflow dynamically updates config/project-scratch-def.json using jq:

YAML / Bash 
- name: Inject Admin Email into Scratch Org Definition
  if: ${{ inputs.adminEmail != '' }}
  run: |
    jq --arg email "${{ inputs.adminEmail }}" '.adminEmail = $email' config/project-scratch-def.json > temp_def.json
    mv temp_def.json config/project-scratch-def.json

Generated Credentials Markdown File

Upon completion, the workflow generates scratch_org_credentials.md and commits it directly back to the active feature branch content with [skip ci]:

Troubleshooting & FAQ Guide

Issue: ProblemDeployingSettings (EinsteinGpt: Not available for deploy) ▼

Cause: Developer Edition Dev Hub orgs do not support einsteinGptSettings in scratch org definition files.

Fix: Remove einsteinGptSettings from config/project-scratch-def.json.

Issue: 'utf-8' codec can't decode byte 0x80 in position 0 ▼

Cause: In headless Linux CI environments, CumulusCI requires an explicit 16-character encryption key for keychain storage.

Fix: Pass CUMULUSCI_KEY: "1234567890abcdef" in workflow environment variables.

Issue: INVALID_AUTH_HEADER / Expired session during org setup ▼

Cause: Generating a password mid-flow invalidates active OAuth tokens.

Fix: Set set_password: false under orgs.scratch.dev in cumulusci.yml and generate the password after deployment completes.

Issue: Duplicate Scratch Orgs created in a single run ▼

Cause: Calling cci org info dev in a separate script step evaluates dev as an unbuilt scratch org config, triggering a 2nd creation.

Fix: Use sf org display --target-org "$SCRATCH_USERNAME" to query org details without creating extra orgs.
